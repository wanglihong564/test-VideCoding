# Prelaunch Test Audit Prompt Reference

Use these Chinese prompts when the user wants reusable copy-paste wording. Adapt names and scope to the current project.

## 0. Choose Audit Mode

请先按上线前审计模式工作，默认只读审计，不要改代码。请先检查项目、测试、配置、部署、数据库迁移、安全和第三方集成，列出问题、证据、风险级别和建议方案。只有在我明确确认“开始修复”后，你才可以进入修复模式；只有在我明确要求“补测试”后，你才可以进入测试编写模式。如果有可用的只读静态扫描脚本，可以先用它收集线索，但不要把脚本结果当成最终结论。

## 1. Identify High-Risk Features

请按业务重要性列出我当前项目里哪些功能必须重点测试，哪些可以暂时不测。请逐个说明：这个功能一旦出错会造成什么后果？它影响的是资金、权限、数据、可用性、用户信任，还是只是展示体验？不要追求全面覆盖，先帮我锁定真正高风险的部分，把清单列给我。先不要修改代码。

## 2. Design Functional Test Cases

请为高风险清单上的每个关键功能设计测试用例，覆盖三类情况：第一，正常流程且业务结果正确；第二，边界值和并发临界，比如 0、负数、超大值、最后一个库存、重复提交、多人同时操作；第三，异常与非法输入，比如未登录、越权、格式错误、缺字段、多字段、过期状态。请重点验证金额、库存、权限、状态、数据库记录等业务结果是否正确，不要只看接口有没有返回成功。

## 3. Build Automated Regression Tests

请把已经确认的重要测试做成可以用一条命令自动运行的回归测试。告诉我这条命令是什么，跑完之后怎么看通过还是失败，哪些关键风险已经覆盖，哪些还没有覆盖。以后每次新增或修改功能，都要重新运行全部测试并汇报结果。

## 4. Prove Tests Are Real

请检查当前测试是不是真实有效。请明确说明每个重要测试到底验证了什么业务结果，有没有真正调用被测功能。然后在安全的本地工作区里故意制造一个小错误，比如把金额、权限判断、库存扣减或状态更新改错，再运行对应测试，确认它会失败。确认后立刻撤回这个故意错误。如果代码改错了测试还通过，说明这个测试无效，必须重写。

## 5. Security Audit

请你现在不要扮演开发者，而是扮演专业测试和安全工程师，对我的后端接口做一次授权范围内的上线前安全审计。请分两步：先审查后端代码，从源头找漏洞；再在本地或我明确授权的测试环境中，用可控请求实测。动态安全测试默认只在本地或测试环境执行；如果目标是生产环境，只允许做低风险验证，除非我明确授权具体测试范围。重点检查：

- 把订单号、用户 ID、资源 ID 改成别人的，看能不能查到、改掉、删掉别人的数据
- 用普通用户身份调用管理员接口，看能不能越权
- 不登录、伪造、篡改、过期 token 或 cookie 后调用接口，看能不能绕过认证
- 在请求里多塞字段，比如 role、isAdmin、ownerId、price、amount，看后端是否错误接受
- 构造 SQL 注入、命令注入、脚本注入、路径穿越等恶意输入
- 完全绕过前端页面，直接请求接口，看前端校验是否被后端重新执行
- 检查返回内容是否泄露密码、token、密钥、内部错误、他人手机号或邮箱
- 对登录、短信、邮件、下单、上传、搜索、导出等接口做低强度防刷检查

请把发现的问题列成清单，逐条标明风险高低、后果、复现方式、请求/响应证据和建议修复方案。注意：先不要修改代码。

## 6. Launch Engineering Review

请补充检查上线工程风险：环境变量、生产/测试环境差异、HTTPS、CORS、Cookie、域名和回调地址、数据库备份、迁移脚本、回滚方案、构建命令、启动命令、健康检查、缓存刷新、日志、监控、告警、错误追踪、第三方支付/短信/邮箱/OAuth/对象存储/webhook 的成功、失败、重复回调和超时重试。请列出阻塞项、高风险项和可以延后的项。

## 7. Supply Chain Review

请检查依赖和构建链风险：项目是否有正确的锁文件，比如 package-lock.json、pnpm-lock.yaml、yarn.lock、poetry.lock、uv.lock、requirements.txt、Cargo.lock、go.sum 等；能否运行当前生态对应的安全扫描，比如 npm audit、pnpm audit、pip-audit、cargo audit 或项目已有的安全检查命令；Dockerfile、compose、CI 脚本、构建产物里有没有密钥、调试配置、不安全安装步骤或未固定版本的基础镜像。请把无法运行的检查也列出来，并说明原因。

## 8. Classify Severity And Evidence

请给每个发现的问题标严重级别：P0 Blocker、P1 High、P2 Medium、P3 Low。P0 包括资金算错、越权、数据丢失或损坏、生产密钥泄露、无法回滚、生产无法启动；P1 包括核心流程失败、重要安全缺口、迁移风险、支付或 webhook 失败、关键监控缺失。每个 P0、P1 和安全问题必须写清楚影响范围、复现步骤、实际结果、期望结果、证据位置、建议修复方案，以及是否阻塞上线。

## 9. Reduce Over-Testing

请审查当前测试是否存在过度测试：有没有为不重要的展示功能写了大量测试？有没有重复、脆弱、没有业务意义、只测 mock 的用例？请优先保证资金、权限、数据、可用性相关功能的测试深度，其余次要测试帮我精简。判断标准是业务风险，不是测试数量。

## 10. Final Go/No-Go Decision

请基于以上检查给出明确上线结论：Go、Conditional Go 或 No Go。判定规则是：只要有未解决的 P0，就必须 No Go；有未确认的 P1，通常是 Conditional Go 或 No Go；只有没有未解决的 P0、没有未接受的 P1、核心测试通过、回滚/备份/环境/监控/关键第三方路径可接受时，才可以 Go。请列出阻塞项、高风险项、已通过的测试、供应链检查结果、未覆盖的风险、上线前必须人工确认的事项，以及如果上线后出问题应该如何回滚。
